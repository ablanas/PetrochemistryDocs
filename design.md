# 石油化学MOD
 
## リアル加工設計書 v9.3
 
*Minecraft 1.21.1 / NeoForge | 実際の石油化学プロセス準拠*
 
---
 
# 1. 設計思想
 
本設計書では、実際の石油精製・石油化学プロセスをMinecraftゲームプレイに落とし込むことを目標とします。以下の実在プロセスを参考に、ゲームバランスを保ちながらリアルな加工チェーンを実装します。
 
- 常圧蒸留（CDU）→ 減圧蒸留（VDU）の2段階精製
- 流動接触分解（FCC）・水素化分解（HCC）による重質油の軽質化
- 接触改質（Catalytic Reforming）によるオクタン価向上とBTX生産
- 水蒸気分解（Steam Cracking）によるオレフィン製造
- 重合・共重合によるポリマー製造
- 脱硫（HDS）プロセスによる硫黄除去

## 2.9 バニラアイテム製造レシピ上書き

このMODはバニラのかまど製錬レシピを**無効化**し、より現実的な工程に置き換えます。対象アイテムはかまどでは製錬できなくなります。序盤プレイのために、鉄・銅については「簡易精錬炉」（バニラかまどサイズのMODブロック）で粗製インゴット（効率50%）を得る代替ルートも用意します。

### 2.9.1 鉄インゴット（iron_ingot）

**バニラ上書き**：鉄鉱石 → かまど → 鉄インゴット を**無効化**
| 工程 | 機械 | 入力 | 出力 | 条件 |
|---|---|---|---|---|
| 1 | 破砕機（3x4x3） | 鉄鉱石 × 1 | 鉄鉱石粉 × 2 | RF消費 |
| 2 | 焙焼炉（4x7x4） | 鉄鉱石粉 × 2 | Fe2O3粉末 × 2 + SO2（硫化鉄の場合） | 600℃ / 自己発熱 |
| 3 | 電気アーク炉（EAF） | Fe2O3粉末 × 4 + 石炭 × 1 | 鉄インゴット × 4 | 1550℃ / 炭素還元 / 黒鉛電極消耗 |

**序盤代替（簡易精錬炉）**：鉄鉱石 × 1 → 鉄インゴット × 1（収率50%。スラグ廃棄物発生）

# 6. 加工フロー全体図

## 6.1 鉱石 → 触媒 → 精製の全体チェーン

```
[採掘]
 └─鉱石群 → 破砕機 → ボールミル（スラリー化）
               └─ 浮遊選鉱機（精鉱分離）
                   └─ 焙焼炉（酸化）→ 湿式浸出槽（浸出）→ 晶析装置（結晶化）
                                                          → 電解精錬槽（金属精製）

[中間素材製造（EAF系）]
 ├─ EAF：ニッケル板 / インコネル管 / ステンレス管バンドル / ステライト棒 / フェロクロム
 ├─ 触媒調製炉：セラミックファイバーブランケット（断熱キャスタブル原料）
 └─ クラフト台：精密バルブ / 熱電対素線 / 耐火モルタル

[触媒製造]
 ├─ ゼオライト系（FCC触媒）：オートクレーブ → 遠心分離 → イオン交換塔 → 噴霧乾燥機
 ├─ Al2O3担体系（HDS/改質触媒）：回転焼成炉 → 含浸ユニット × 複数回 → 触媒調製炉
 ├─ α-Al2O3担体系（SMR触媒）：回転焼成炉（1400℃）→ 含浸ユニット → 還元焼成炉
 └─ MgCl2担体系（ZN触媒）：噴霧乾燥機 → 含浸ユニット × 複数回

[水素製造]
 └─ SMR（LPG+水蒸気）→ PSA（精製）→ 高純度水素 → HDS・還元焼成炉

[石油精製]
 └─ 原油 → 原油加熱炉 → CDU → VDU → FCC / HDS / Reformer / Cracker
                                             ↓
                                        BTX抽出 → PX製造
                                             ↓
                                        重合炉 → ポリマー製品
```

# 8. 技術実装メモ

## 8.1 流体品質システム（オプション）

各流体にNBTタグで「硫黄含量」「オクタン価」「粘度グレード」を持たせると、より本格的になります。例えば脱硫前の軽油をそのまま機械に使うと機械が劣化するなどの実装が可能です。

## 8.2 触媒劣化システム

- 触媒はBlockEntityのNBTに残存活性度（0-100%）を保存
- 活性度が下がると処理効率・収率が低下
- FCCはコーク焼成で自動再生、他は触媒アイテム補充で回復
- SMR触媒は硫黄被毒で不可逆失活（0%になると交換必須）

## 8.3 温度・圧力システム

- 各装置はBlockEntityに「現在温度」を保持
- 起動後、燃料消費しながら設定温度まで昇温（昇温中は効率低下）
- 急冷が必要な装置（スチームクラッカー等）は急冷装置未接続で収率ペナルティ
- 回転焼成炉は温度プロファイル設定（昇温速度・保持時間・冷却速度）をGUIで管理

## 8.4 ボイラー・蒸気システム

- 複数装置で共用の蒸気動力を採用（VDU・BTX抽出・重合炉・化学品蒸留塔等）
- 蒸気はSteamFluidとして実装、専用蒸気パイプで輸送
- ボイラー燃料：重質軽油・常圧残渣・減圧残渣

## 8.5 マルチブロック実装方針

- MultiblockValidatorクラスで構造チェック（形状をJSON定義）
- 完成/未完成をBlockStateで視覚的に区別
- コアBlockEntityがネットワーク全体を管理
- マルチブロック定義ファイルはデータパック対応（data/modid/multiblocks/*.json）とし、後からアドオンで拡張可能にする
- コントローラーブロックを右クリックするとホログラム表示が起動し、マルチブロックが完成していない間は常時表示を維持する（完成後に自動消灯。再クリックで手動消灯）

## 8.6 マルチブロック ホログラム表示システム

マルチブロック構造の組み立てガイドをプレイヤーに視覚的に伝えるため、クライアントサイドのホログラムレンダリングシステムを実装します。NeoForge 1.21.1 のレンダリングAPI（RenderLevelStageEvent・PoseStack・VertexConsumer）を使用します。

### 8.6.1 ブロック状態の4色分類

| 状態 | 色（RGBA） | Enum定数 | 説明 |
| --- | --- | --- | --- |
| 配置済み | 緑 (0,255,64,80) | PLACED | 正しいブロックが正しい位置にある（完了） |
| 不足 | 赤 (255,0,32,100) | MISSING | 未配置。パルス点滅で強調 |
| 誤ブロック | オレンジ (255,160,0,100) | WRONG_BLOCK | 異なるブロックが配置されている |
| I/Oポート | シアン (0,220,255,120) | IO_PORT | 流体入出力口・アイテム搬出入口。回転矢印付き |

### 8.6.2 レンダリングアーキテクチャ

- HologramRenderer（クライアント専用）：RenderLevelStageEvent.Stage.AFTER_TRANSLUCENT_BLOCKS にフックし半透明キューへ描画。デプスバッファ書き込み無効化（GL_DEPTH_WRITEMASK=false）で壁越し表示を実現
- BlockOutlineRenderer：各ブロック位置に1.02倍スケールの立方体ワイヤーフレームを描画（RenderType.lines()使用）
- GhostBlockRenderer：ブロックモデルをBufferBuilderで取得しalpha=0.35で描画
- MultiblockHologramData（パケット）：サーバーが構造チェック結果をList<BlockPos, HologramState>としてクライアントへ送信。チェック頻度は20tickごと

### 8.6.3 アニメーション仕様

- パルス点滅（MISSING）：alpha = 0.2 + 0.15 * sin(tickTime * 0.1 * PI)
- 回転矢印（IO_PORT）：Y軸周りをtickTime * 2°/tickで回転
- 加熱波形（稼働中）：装置昇温中にブロック上面からゆらぎパーティクルを4tickごと放出
- 完成フラッシュ：全ブロックがPLACEDになった瞬間、緑→白→消灯の0.5秒フラッシュ

### 8.6.4 ステータスHUDオーバーレイ

コントローラーを視点に捉えている間、画面左下に以下をオーバーレイ表示（RenderGuiEvent.Postにフック）：

- 装置名・稼働状態（IDLE / HEATING / RUNNING / ERROR）
- 現在温度/目標温度（℃）・処理進捗バー（0-100%）
- 燃料残量（mB）・触媒活性度（%）・エラーコード（未完成ブロック名一覧）
- クライアント設定でオフ可能。デフォルトはオン

### 8.6.5 パフォーマンス最適化

- 頂点バッファはキャッシュし構造変更がない限り毎フレーム再構築しない（ダーティフラグ管理）
- 視錐台カリング：カメラ視野外はスキップ
- 同時表示ホログラム数は最大4マルチブロックに制限
